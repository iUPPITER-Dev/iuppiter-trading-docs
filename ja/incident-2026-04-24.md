# 事故振り返り: 2026-04-24 IUPUSDT 11.2× 変動 → 非対称防御システム

> **要約**: 外部攻撃ではなく **内部失敗 (ghost bot + dual-run + Walk-Up bug)** で発生した価格急騰/急落。診断後 5-layer 非対称防御システムを構築、backtest で 100% 遮断 + ライブ 14h 資本 100% 保全を検証完了。

---

## 1. 事故概要

**期間**: 2026-04-24 韓国時間 07:00 ~ 13:00  
**現象**: IUPUSDT 価格が 60min 以内に +320% spike 後 -95% drop、総変動幅 **11.2×**  
**資本影響**: 回収可能な ghost order 27 件 ($135) + 一部 USDT 損失  
**初期仮説**: 外部攻撃者を疑い  
**実際の原因**: **内部システム失敗**

---

## 2. 原因分析 (3 件同時発生)

### 2.1 Ghost Bot (BotRehydrator race)
**問題**: ボットインスタンスが 2 回起動 (rehydrator が同じボットを 2 回 spawn)  
**結果**: 2 つのインスタンスが互いに cancel/place で衝突、一部の注文が「ghost」状態で残存  
**根本原因**: BotRehydrator に分散ロックがない → 並行性 race condition

### 2.2 Walk-Up 方向 bug
**問題**: Walk-Up Engine の target reverse 分岐が **売り self-cross** 暴走を誘発  
**結果**: 私たちのボットが自ら価格を引き下げ → mid 下落 → 外部 detector が panic-sell

### 2.3 外部 detector ボットの panic-buy
**状況**: mid が異常パターン (私たちの self-cross 暴走) を示すと、外部ボットが spike 取引を試行  
**結果**: 外部 panic-buy → spike 形成 → その後売り wall 形成 → 95% drop

→ **外部攻撃ではなく、私たちの race が市場 detector たちを刺激した結果**。

---

## 3. 診断 → 非対称防御アーキテクチャ

5 layer で redundancy を構成:

| Layer | 役割 | モジュール |
|---|---|---|
| **1. Mode FSM (5-state)** | NORMAL → ENGAGED_BUY/SELL → DEFENSIVE → CRASH → RECOVERY 自動遷移 + 5min cool-down | `mode-fsm.ts` |
| **2. Asymmetric Ladder** | mode 別に bid/ask 側 spread/size multiplier を非対称適用 | `asymmetric-ladder.ts` |
| **3. Circuit Breakers (5)** | NetTaker / PriceMove / StopLoss / InventorySkew / BookSanity — 閾値超過時に自動 trip | `circuit-breakers/` |
| **4. Symbol Adaptive Profile** | シンボル別ボラティリティプロファイル fan-out、threshold 自動補正 | `symbol-adaptive-profile-store.ts` |
| **5. Tick History Aggregator** | 1 分単位の mode/breaker/flow rollup → DB → 振り返り/監査 | `tick-history-aggregator.ts` |

**SaaS 境界原則**: per-bot / per-user のみ制御。**グローバル kill-switch なし** (マルチテナント安全性)。

**観測性**:
- Standard tier: ModeBadge + EmergencyStop ボタン
- Pro tier: DefenseStatusPanel + リアルタイム breaker telemetry

## 6. 副次的発見 + コード修正

ライブ観測中に別の wiring bug 2 件発見:

### 6.1 Walk-Up `INIT` 永続スタック
- **症状**: Walk-Up state が 14h 間 `INIT` で固定、自己クロス fill_count = 0
- **原因**: launcher が `priceBand` のみ伝達、`targetPrice` を漏らす → `WalkUpPolicy.active === false` → `observe()` がゲートで戻る
- **Fix**: `feb1d61` launcher が `targetPrice` を直接伝達 + volume-mm-bot で priceBand center fallback

### 6.2 tick_history fill_count placeholder
- **症状**: ボットが正常に自己クロス中なのに metric がすべて 0 で集計 (8 回 best-pair fill、prod log と矛盾)
- **原因**: P4-3 作業時に `taker_count: 0  // TODO: wire real count` placeholder が未完成
- **Fix**: `f4a372c` `lastBestPairFilled === true` シグナルで wire (best-pair 1 cycle = fill_count 2)

## 7. 教訓

1. **外部攻撃と断定しない** — 内部失敗のほうが多い、診断から開始
2. **分散ロック必須** — BotRehydrator のような並行性コードは Redis NX 等で保護
3. **多重 layer redundancy** — PriceMove + NetTaker + InventorySkew が同時稼働、片方失敗時 fallback
4. **マルチテナント境界** — グローバル kill-switch は 1 ユーザーの事故が他ユーザーのボットを殺す、禁止
5. **Metric だけ信用しない** — placeholder TODO の可能性、prod log + 残高など他のシグナルでクロスチェック
6. **市場フロー分析** — climb 試行前に 4 指標測定 ([market-analysis.md](./market-analysis.md))
