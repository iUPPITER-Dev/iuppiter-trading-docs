# 사고 회고: 2026-04-24 IUPUSDT 11.2× 변동 → 비대칭 방어 시스템

> **요약**: 외부 공격이 아닌 **내부 실패 (ghost bot + dual-run + Walk-Up bug)** 로 발생한 가격 폭등/폭락. 진단 후 5-layer 비대칭 방어 시스템 구축, backtest 100% 차단 + 라이브 14h 자본 100% 보존 검증 완료.

## 2. 원인 분석 (3 가지 동시 발생)

### 2.1 Ghost Bot (BotRehydrator race)
**문제**: 봇 인스턴스가 2회 가동 (rehydrator 가 동일 봇을 두 번 spawn)  
**결과**: 두 인스턴스가 서로 cancel/place 충돌, 일부 주문이 "고스트" 상태로 잔존  
**근본 원인**: BotRehydrator 에 분산 lock 부재 → 동시성 race condition

### 2.2 Walk-Up 방향 bug
**문제**: Walk-Up Engine 의 target reverse 분기가 **매도 self-cross** 폭주를 유발  
**결과**: 우리 봇이 자체적으로 가격을 끌어내림 → mid 하락 → 외부 detector 가 panic-sell

### 2.3 외부 detector 봇의 panic-buy
**상황**: mid 가 비정상 패턴 (우리 self-cross 폭주) 으로 보이자 외부 봇이 spike 거래 시도  
**결과**: 외부 panic-buy → spike 형성 → 이후 매도 wall 형성 → 95% drop

→ **외부 공격이 아니라, 우리의 race 가 시장 detector 들을 자극한 결과**.

---

## 3. 진단 → 비대칭 방어 아키텍처

5 layer 로 redundancy 구성:

| Layer | 역할 | 모듈 |
|---|---|---|
| **1. Mode FSM (5-state)** | NORMAL → ENGAGED_BUY/SELL → DEFENSIVE → CRASH → RECOVERY 자동 전이 + 5min cool-down | `mode-fsm.ts` |
| **2. Asymmetric Ladder** | mode 별 bid/ask side spread/size multiplier 비대칭 적용 | `asymmetric-ladder.ts` |
| **3. Circuit Breakers (5)** | NetTaker / PriceMove / StopLoss / InventorySkew / BookSanity — 임계 초과 시 자동 trip | `circuit-breakers/` |
| **4. Symbol Adaptive Profile** | 심볼별 변동성 프로파일 fan-out, threshold 자동 보정 | `symbol-adaptive-profile-store.ts` |
| **5. Tick History Aggregator** | 1분 단위 mode/breaker/flow rollup → DB → 회고/감사 | `tick-history-aggregator.ts` |

**SaaS 경계 원칙**: per-bot / per-user 만 통제. **글로벌 kill-switch 없음** (멀티테넌트 안전성).

**관측성**:
- Standard tier: ModeBadge + EmergencyStop 버튼
- Pro tier: DefenseStatusPanel + 실시간 breaker telemetry

## 6. 부수 발견 + 코드 수정

라이브 관측 중 별도 wiring bug 2건 발견:

### 6.1 Walk-Up `INIT` 영원히 stuck
- **증상**: Walk-Up state 14h 동안 `INIT` 고정, 자전거래 fill_count = 0
- **원인**: launcher 가 `priceBand` 만 전달, `targetPrice` 누락 → `WalkUpPolicy.active === false` → `observe()` 가 첫 호출에서 빠져나감
- **Fix**: `feb1d61` launcher targetPrice 직접 전달 + volume-mm-bot 에 priceBand center fallback

### 6.2 tick_history fill_count placeholder
- **증상**: 봇이 정상 자전거래 중인데 metric 모두 0 으로 집계 (8회 best-pair fill, prod log 와 모순)
- **원인**: P4-3 작업 시 `taker_count: 0  // TODO: wire real count` placeholder 가 미완성으로 남음
- **Fix**: `f4a372c` `lastBestPairFilled === true` 신호로 wire (best-pair 1 cycle = fill_count 2)

## 7. 교훈

1. **외부 공격으로 단정하지 말 것** — 내부 실패가 더 흔함, 진단부터 시작
2. **분산 lock 필수** — BotRehydrator 같은 동시성 코드는 Redis NX 등으로 보호
3. **다중 layer redundancy** — PriceMove + NetTaker + InventorySkew 동시 작동, 한쪽 실패 시 fallback
4. **Multi-tenant 경계** — 글로벌 kill-switch 는 한 user 의 사고가 다른 user 의 봇을 죽임, 금지
5. **Metric 만 신뢰하지 말 것** — placeholder TODO 가능성, prod log + 잔고 등 다른 신호로 cross-check
6. **시장 흐름 분석** — climb 시도 전 4 지표 측정 ([market-analysis.md](./market-analysis.md))
