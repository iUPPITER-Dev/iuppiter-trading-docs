# 事故复盘: 2026-04-24 IUPUSDT 11.2× 波动 → 非对称防御系统

> **概要**: 由 **内部失败 (ghost bot + dual-run + Walk-Up bug)** 引起的非外部攻击的价格暴涨/暴跌。诊断后构建 5-layer 非对称防御系统,backtest 100% 阻断 + 实时 14h 资本 100% 保护验证完成。

---

## 1. 事故概要

**期间**: 2026-04-24 韩国时间 07:00 ~ 13:00  
**现象**: IUPUSDT 价格在 60min 内 +320% spike 后 -95% drop,总波动幅度 **11.2×**  
**资本影响**: 可回收 ghost order 27 件 ($135) + 部分 USDT 损失  
**初步假设**: 怀疑外部攻击者  
**实际原因**: **内部系统失败**

---

## 2. 原因分析 (3 件同时发生)

### 2.1 Ghost Bot (BotRehydrator race)
**问题**: 机器人实例启动 2 次 (rehydrator 对同一机器人 spawn 2 次)  
**结果**: 两个实例在 cancel/place 上互相冲突,部分订单残留为 "ghost" 状态  
**根本原因**: BotRehydrator 缺少分布式锁 → 并发 race condition

### 2.2 Walk-Up 方向 bug
**问题**: Walk-Up Engine 的 target reverse 分支引发 **抛售 self-cross** 失控  
**结果**: 我们的机器人自行将价格推下 → mid 下跌 → 外部 detector panic-sell

### 2.3 外部 detector 机器人的 panic-buy
**情况**: mid 显示异常模式 (我们的 self-cross 失控) 时,外部机器人尝试 spike 交易  
**结果**: 外部 panic-buy → spike 形成 → 之后形成抛售 wall → 95% drop

→ **非外部攻击,而是我们的 race 刺激了市场 detector 们的结果**。

---

## 3. 诊断 → 非对称防御架构

5 layer 构成 redundancy:

| Layer | 角色 | 模块 |
|---|---|---|
| **1. Mode FSM (5-state)** | NORMAL → ENGAGED_BUY/SELL → DEFENSIVE → CRASH → RECOVERY 自动转移 + 5min cool-down | `mode-fsm.ts` |
| **2. Asymmetric Ladder** | 按 mode 对 bid/ask 侧 spread/size multiplier 非对称应用 | `asymmetric-ladder.ts` |
| **3. Circuit Breakers (5)** | NetTaker / PriceMove / StopLoss / InventorySkew / BookSanity — 阈值超过时自动 trip | `circuit-breakers/` |
| **4. Symbol Adaptive Profile** | 按 symbol 波动性 profile fan-out,threshold 自动校准 | `symbol-adaptive-profile-store.ts` |
| **5. Tick History Aggregator** | 1 分钟单位 mode/breaker/flow rollup → DB → 回顾/审计 | `tick-history-aggregator.ts` |

**SaaS 边界原则**: 仅 per-bot / per-user 控制。**无全局 kill-switch** (多租户安全性)。

**可观测性**:
- Standard tier: ModeBadge + EmergencyStop 按钮
- Pro tier: DefenseStatusPanel + 实时 breaker telemetry

## 6. 副发现 + 代码修正

实时观察中发现另外的 wiring bug 2 件:

### 6.1 Walk-Up `INIT` 永久 stuck
- **症状**: Walk-Up state 14h 内固定为 `INIT`,自交易 fill_count = 0
- **原因**: launcher 仅传递 `priceBand`,遗漏 `targetPrice` → `WalkUpPolicy.active === false` → `observe()` 在 gate 处返回
- **Fix**: `feb1d61` launcher 直接传递 `targetPrice` + volume-mm-bot 中 priceBand center fallback

### 6.2 tick_history fill_count placeholder
- **症状**: 机器人正常自交易但 metric 全部聚合为 0 (8 次 best-pair fill,与 prod log 矛盾)
- **原因**: P4-3 工作时 `taker_count: 0  // TODO: wire real count` placeholder 未完成
- **Fix**: `f4a372c` 用 `lastBestPairFilled === true` 信号 wire (best-pair 1 cycle = fill_count 2)

## 7. 教训

1. **不要断定为外部攻击** — 内部失败更常见,从诊断开始
2. **分布式锁必需** — BotRehydrator 等并发代码必须用 Redis NX 等保护
3. **多层 redundancy** — PriceMove + NetTaker + InventorySkew 同时工作,一方失败时 fallback
4. **多租户边界** — 全局 kill-switch 让一个用户的事故杀死其他用户的机器人,禁止
5. **不要仅信 metric** — placeholder TODO 可能,与 prod log + 余额等其他信号 cross-check
6. **市场流向分析** — climb 尝试前测量 4 指标 ([market-analysis.md](./market-analysis.md))
